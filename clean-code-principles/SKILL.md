# Clean Code Principles

## Цель
Задать единый стандарт чистого кода для всей кодовой базы, чтобы улучшить читаемость, поддержку и надежность.

## Когда использовать
Используй этот skill, если нужно:
- написать новый код;
- провести рефакторинг существующего кода;
- сделать PR-ревью по качеству;
- разрешить спор по стилю и структуре решения.

## Обязательные правила
1. Код должен быть понятен по именам и структуре без чтения комментариев.
2. Один класс и один метод должны иметь одну ответственность.
3. Не использовать общий `Exception` для ожидаемых бизнес-сценариев.
4. Не допускать скрытых побочных эффектов и неявных зависимостей.
5. Длина строки не должна превышать 120 символов.
6. Логи и сообщения ошибок должны быть на английском языке.

## Рекомендуемые практики
1. Применять KISS: выбирать самое простое решение, которое покрывает требования.
2. Применять DRY для стабильных повторов.
3. Применять YAGNI: не добавлять абстракции без реального сценария.
4. Использовать guard clauses для уменьшения вложенности.
5. Декомпозировать сложные методы на небольшие смысловые шаги.
6. Пробрасывать `CancellationToken` в async-цепочке.

## Анти-паттерны
1. God class с множеством несвязанных обязанностей.
2. Методы, которые одновременно валидируют, вычисляют и сохраняют данные.
3. Нечитаемые сокращения в именах типов, методов и переменных.
4. Скрытая мутация состояния без явного контракта.

## Алгоритм принятия решений
1. Определи ответственность текущего класса и метода.
2. Убери лишние обязанности из одного места в отдельные компоненты.
3. Проверь имена на ясность и доменный смысл.
4. Сократи сложность ветвлений и вложенности.
5. Убедись, что поведение тестируемо и предсказуемо.

## Проверка перед завершением
1. Код читается без расшифровки намерений.
2. Нет нарушения SRP на уровне методов и классов.
3. Нет неявных побочных эффектов.
4. Нет строк длиннее 120 символов.
5. Сообщения логов и ошибок соответствуют стандарту языка.
