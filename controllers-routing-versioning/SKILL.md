# Controllers Routing Versioning

## Цель
Обеспечить предсказуемый routing и безопасное evolution API без поломки клиентов.

## Когда использовать
Используй этот skill, если нужно:
- менять URL-маршруты контроллера;
- менять семантику endpoint;
- определить, breaking change это или нет;
- принять решение о создании новой версии API.

## Обязательные правила
1. Breaking change должен выпускаться только в новой версии API.
2. Не менять поведение опубликованной версии, если это ломает клиентов.
3. Маршруты новой версии должны быть единообразными в рамках ресурса.
4. Route parameters и HTTP-методы должны однозначно отражать действие.
5. Для одинаковых сценариев использовать одинаковые подходы роутинга.

## Рекомендуемые практики
1. Для новых версий использовать resource-style маршруты.
2. Держать маршруты короткими и без action-слов, где это возможно.
3. Поддерживать явную migration strategy между версиями.
4. При сомнении классифицировать изменение как breaking.

## Анти-паттерны
1. Тихая замена маршрута в существующей версии.
2. Смешение несовместимых route-style в одной версии.
3. Изменение обязательных полей запроса без новой версии.

## Примеры

### Breaking
- Переименование `PATCH /users/{id}/email` в `PATCH /users/{id}/contacts/email`.
- Изменение типа поля `userId` из `Guid` в `string`.

### Non-breaking
- Добавление нового необязательного поля в response DTO.
- Добавление нового endpoint без изменения старых.

## Алгоритм принятия решений
1. Зафиксируй текущий маршрут и контракт endpoint.
2. Оцени изменение по критерию client impact.
3. Если клиент может сломаться, создавай новую версию.
4. Обнови маршруты и документацию для новой версии.
5. Проверь, что старая версия остается рабочей.

## Проверка перед завершением
1. Breaking изменения не попали в старую версию.
2. Маршруты в новой версии согласованы между собой.
3. Стратегия миграции между версиями описана и протестирована.
