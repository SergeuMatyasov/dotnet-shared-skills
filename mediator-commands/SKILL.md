# Mediator Commands

## Цель
Сделать обработку команд через MediatR предсказуемой, безопасной и удобной для сопровождения.

## Когда использовать
Используй этот skill, если нужно:
- добавить новую Command и Handler;
- изменить контракт существующей команды;
- провести ревью командного pipeline;
- проверить корректность побочных эффектов и транзакций.

## Обязательные правила
1. Команда должна описывать намерение изменения состояния в business-терминах.
2. Обработчик команды должен иметь одну ответственность: оркестровать сценарий изменения состояния.
3. Валидация входных данных должна выполняться до побочных эффектов.
4. Побочные эффекты должны быть явными: запись в БД, отправка событий, вызовы внешних сервисов.
5. Для ошибок использовать доменные/прикладные исключения с понятным контекстом.
6. В async-операциях обязательно пробрасывать `CancellationToken`.
7. Запрещено размещать HTTP-специфику в командах и обработчиках.
8. Команда не должна возвращать избыточные данные: только то, что нужно вызывающей стороне.

## Рекомендуемые практики
1. Называть команды в формате действия: `CreateUserCommand`, `UpdateUserEmailCommand`.
2. Использовать immutable DTO/record для входных данных команды.
3. Выделять бизнес-правила в доменные сущности или отдельные сервисы, а не перегружать Handler.
4. Для критичных сценариев предусматривать идемпотентность.
5. Для составных сценариев явно описывать порядок шагов в Handler.
6. Логировать ключевые этапы с минимально достаточным контекстом.

## Анти-паттерны
1. Толстый Handler с бизнес-логикой, маппингом, валидацией и интеграциями в одном методе.
2. Скрытые побочные эффекты, неочевидные из имени команды.
3. Использование команды как Query (без изменения состояния).
4. Проглатывание исключений и возврат `false` без объяснения причины.
5. Изменение нескольких агрегатов без продуманной транзакционной границы.

## Примеры

### Do
```csharp
public sealed record UpdateUserEmailCommand(Guid UserId, string Email) : IRequest<Unit>;

public sealed class UpdateUserEmailCommandHandler(IUsersRepository usersRepository)
	: IRequestHandler<UpdateUserEmailCommand, Unit>
{
	public async Task<Unit> Handle(UpdateUserEmailCommand request, CancellationToken ct)
	{
		ArgumentNullException.ThrowIfNull(request);

		var user = await usersRepository.GetByIdAsync(request.UserId, ct);
		user.UpdateEmail(request.Email);
		await usersRepository.SaveChangesAsync(ct);

		return Unit.Value;
	}
}
```

### Don't
```csharp
public async Task<bool> Handle(UpdateUserEmailCommand request, CancellationToken ct)
{
	// HTTP status mapping + validation + business rules + external calls + save in one place
	// swallow exceptions and return false
	return false;
}
```

## Алгоритм принятия решений
1. Определи изменение состояния, которое инициирует команда.
2. Спроектируй минимальный контракт команды.
3. Определи правила валидации и точку их выполнения.
4. Зафиксируй транзакционную границу и побочные эффекты.
5. Реализуй Handler как оркестратор сценария.
6. Добавь обработку ошибок и структурные логи.
7. Проверь идемпотентность для повторных вызовов, если это важно для сценария.

## Проверка перед завершением
1. Команда действительно меняет состояние.
2. Контракт команды минимален и понятен.
3. Валидация выполняется до побочных эффектов.
4. `CancellationToken` проброшен во все async-вызовы.
5. Побочные эффекты и транзакционность выражены явно.
6. Ошибки и логи помогают расследовать инцидент.
7. Handler не содержит лишней HTTP/UI-специфики.
