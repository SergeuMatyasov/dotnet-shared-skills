## Цель
Писать тесты, которые:
- легко читать;
- безопасно изменять;
- быстро запускать;
- просто расширять без дублирования;
- помогают локализовать причину падения.

---

## Обязательное правило: `summary` на русском для каждого теста
Для каждого нового или измененного тестового метода добавляй XML `summary` на русском прямо над методом.

Требования к `summary`:
- Описывает проверяемое поведение, а не внутреннюю реализацию.
- Кратко и конкретно: 1–2 предложения.
- Без общих формулировок вроде «Проверяет метод».
- Должен быть понятен без чтения тела теста.

Пример:

```csharp
/// <summary>
/// Проверяет, что валидатор возвращает ошибку, когда UserId пустой.
/// </summary>
[Fact]
public async Task ValidateAsync_ShouldReturnError_WhenUserIdIsEmpty()
{
    // Arrange
    // Act
    // Assert
}
```

Для `[Theory]`:
- `summary` описывает общее поведение для всех входных наборов данных.
- Если наборы сильно различаются по смыслу, лучше разделить их на несколько тестов.

Пример:

```csharp
/// <summary>
/// Проверяет, что валидатор отклоняет некорректные значения email во всех поддерживаемых негативных сценариях.
/// </summary>
[Theory]
[InlineData("")]
[InlineData(" ")]
[InlineData("invalid-email")]
public async Task ValidateAsync_ShouldReturnError_WhenEmailIsInvalid(string email)
{
    // Arrange
    // Act
    // Assert
}
```

## Базовые правила написания тестов

### 1. Используй структуру Arrange / Act / Assert
Каждый тест должен быть явно разделен на три смысловые части:
- `Arrange` — подготовка данных и зависимостей;
- `Act` — одно целевое действие;
- `Assert` — одна логическая группа проверок одного сценария.

Правила:
- Не смешивай `Act` и `Assert`.
- Не делай несколько независимых `Act` в одном тесте.
- Если секция отсутствует, это должно быть осознанно и очевидно.

### 2. Именуй тесты по шаблону
Используй формат:

`Method_ShouldExpectedResult_WhenCondition`

Примеры:
- `ValidateAsync_ShouldReturnError_WhenUserIdIsEmpty`
- `Handle_ShouldCreateOrder_WhenCommandIsValid`
- `Map_ShouldReturnNull_WhenSourceIsNull`

Если тест проверяет выброс исключения:
- `Method_ShouldThrowException_WhenCondition`

Пример:
- `Handle_ShouldThrowValidationException_WhenRequestIsInvalid`

### 3. Один тест — один логический сценарий
Допустимо несколько `Assert`, если они описывают один результат одного сценария.

Недопустимо:
- одновременно проверять успешный путь и ошибочный путь;
- проверять несколько независимых бизнес-правил в одном тесте;
- использовать один тест как «мини-интеграционный сценарий», если нужен unit-тест.

### 4. Тест должен быть детерминированным
Запрещено зависеть от:
- текущего времени без фиксации;
- часового пояса машины;
- сети;
- файловой системы;
- случайных значений без фиксированного seed;
- порядка выполнения других тестов.

Правила:
- Время передавай через абстракцию (`IClock`, `TimeProvider`, fake clock).
- GUID и случайные значения фиксируй.
- Не используй реальные таймеры и `Task.Delay`, если можно обойтись без них.

### 5. Изолируй внешние зависимости
Любые внешние зависимости изолируй через:
- моки;
- стабы;
- fake-реализации;
- in-memory варианты, если это соответствует типу теста.

Изолируй:
- API;
- БД;
- кэш;
- брокеры;
- файловое хранилище;
- системное время;
- окружение и конфигурацию.

## Дополнительные правила, которых сейчас не хватает и которые стоит добавить

### 6. Проверяй наблюдаемое поведение, а не детали реализации
Тест должен быть устойчив к рефакторингу.

Хорошо:
- проверка результата;
- проверка изменения состояния;
- проверка публичного контракта;
- проверка значимых вызовов внешних зависимостей.

Плохо:
- проверка внутренних приватных деталей;
- чрезмерная проверка количества вызовов, если это не часть контракта;
- жёсткая привязка к внутреннему порядку операций без необходимости.

Пример плохой практики:
- тест падает после безопасного рефакторинга, хотя поведение не изменилось.

### 7. На один тест — минимально необходимый набор проверок
Проверяй только то, что важно для сценария.

Не нужно:
- в каждом тесте проверять все поля результата;
- дублировать проверки, уже покрытые другими тестами;
- утверждать несущественные детали.

Нужно:
- проверять только значимые последствия конкретного сценария.

### 8. Устраняй «магические значения»
Все неочевидные значения должны быть названы.

Плохо:
```csharp
var command = new CreateUserCommand("abc", 17);
```

Хорошо:
```csharp
const string invalidUserName = "abc";
const int underage = 17;

var command = new CreateUserCommand(invalidUserName, underage);
```

### 9. Не прячь смысл теста в громоздкие helper-методы
Вспомогательный код допустим, но он не должен скрывать бизнес-смысл.

Хорошо:
- builder с понятными значениями по умолчанию;
- фабрика тестовых данных;
- helper для повторяющейся технической подготовки.

Плохо:
- `CreateValidScenario()` без явного понимания, что именно создается;
- глубокая цепочка helper-методов, из-за которой сценарий теста невозможно понять по месту.

### 10. Тест должен падать с понятным сообщением
Используй выразительные ассерты и группируй их так, чтобы причина падения была очевидна.

Предпочтительно:
- `FluentAssertions`;
- осмысленные имена переменных;
- минимум лишних промежуточных преобразований.

Пример:
```csharp
result.Errors.Should().ContainSingle(x => x.Code == ErrorCodes.UserIdRequired);
```

## Рекомендуемые библиотеки и правила их применения

### Базовый стек
- Фреймворк тестов: `xUnit`
- Assertions: `FluentAssertions`
- Моки: `NSubstitute` предпочтительно, `Moq` допустим
- Генерация данных: `AutoFixture`
- Параметризованные данные: `Theory` + `InlineData` / `MemberData` / `ClassData`

### Как использовать библиотеки
- `FluentAssertions` применяй для читаемых и точных проверок.
- `NSubstitute` применяй для зависимостей, где важен контракт взаимодействия.
- `AutoFixture` используй умеренно. Он не должен делать тест менее понятным.
- Для сложных и доменно значимых объектов чаще лучше builder, чем «полностью случайные» данные.

### Если библиотека не подключена
Если нужная библиотека еще не подключена:
- предложи добавить ее;
- либо выполни подключение в рамках задачи, если это уместно;
- не вводи новый стек без необходимости, если в проекте уже есть принятый стандарт.

## Правила по переиспользованию кода

### Обязательно выносить в общие папки
Если код используется в нескольких тестах или потенциально будет переиспользован, выноси его в общие папки и классы.

Рекомендуемая структура тестового проекта:
- `Application/Common/Fixtures` — фикстуры и подготовка окружения
- `Application/Common/Builders` — builders для доменных и DTO объектов
- `Application/Common/Fakes` — fake-реализации и test doubles
- `Application/Common/TestData` — повторно используемые тестовые данные
- `Application/Common/Extensions` — helper-расширения для тестов

### Правила выноса
- Не дублируй `Arrange`-блоки более чем в 2 тестах.
- Общие builders и фабрики должны иметь безопасные значения по умолчанию.
- Значения по умолчанию должны быть валидными, если тест явно не проверяет невалидный сценарий.
- Хелперы не должны скрывать смысл теста.
- Любой builder должен позволять локально переопределить только значимые для теста поля.

Пример подхода:
```csharp
var command = new CreateOrderCommandBuilder()
    .WithUserId(userId)
    .WithItems(items)
    .Build();
```

## Покрытие сценариев: минимальный набор для каждого юзкейса
Для каждого юзкейса, handler, validator или domain service покрывай минимум следующие группы сценариев:

### 1. Позитивный сценарий
- валидный вход;
- ожидаемый успешный результат;
- корректный вызов зависимостей, если это часть контракта.

### 2. Негативные сценарии
- невалидный вход;
- нарушение бизнес-правил;
- отсутствие обязательных данных;
- конфликтные значения.

### 3. Граничные значения
- `null`;
- пустые строки;
- строки из пробелов;
- минимум;
- максимум;
- значения на границе диапазона;
- пустые коллекции;
- коллекции из одного элемента;
- слишком большие наборы данных, если это значимо.

### 4. Ошибки зависимостей
Если зависимость может вернуть ошибку или бросить исключение, тесты должны проверять:
- как ошибка пробрасывается;
- как ошибка маппится;
- как выполняется fallback, если он предусмотрен;
- не происходит ли частичное изменение состояния.

### 5. Параметризованные наборы данных
Используй `[Theory]`, если:
- логика одна и та же;
- различаются только входные значения;
- кейсы удобно читать в одном месте.

Не используй `[Theory]`, если:
- кейсы имеют разный бизнес-смысл;
- для понимания теста нужны разные `Arrange`-сценарии;
- общий тест становится слишком абстрактным.

## Что отдельно добавить для validator tests
Для валидаторов обязательно проверяй:
- что валидный объект проходит валидацию без ошибок;
- что каждая отдельная бизнес-ошибка возвращается корректно;
- что ошибка привязана к нужному полю, если это важно;
- что текст ошибки или код ошибки соответствует контракту;
- что несколько ошибок возвращаются вместе, если это ожидается;
- что `null`-вход обрабатывается явно, если это применимо.

Дополнительно:
- если используется `FluentValidation`, проверяй не только наличие ошибки, но и ее `PropertyName`, `ErrorCode`, а при необходимости и `Severity`.

---

## Что отдельно добавить для integration tests
Хотя основной акцент на unit-тестах, для integration tests стоит зафиксировать отдельные правила:

- Интеграционный тест должен проверять взаимодействие реальных компонентов системы.
- Количество внешних зависимостей должно быть минимально и контролируемо.
- Для БД используй изолированное тестовое окружение.
- Каждый тест должен сам подготавливать данные и сам очищать последствия, либо использовать транзакционную/эфемерную стратегию.
- Нельзя зависеть от результатов запуска других тестов.
- Для integration tests особенно важно разделять быстрые и медленные тесты.

Полезно добавить маркировку:
- категории тестов;
- отдельный pipeline/фильтр запуска;
- явное разделение папок `Unit` / `Integration` / `Validators`.

---

## Что отдельно добавить про моки
Хорошая практика:
- мокать только реальные внешние зависимости;
- не мокать простые DTO, value object и чистые доменные модели;
- не мокать то, что проще создать как обычный объект;
- не проверять каждый вызов без необходимости.

Проверки взаимодействия нужны только если:
- вызов — часть контракта;
- важно, что зависимость не была вызвана;
- важно число вызовов;
- важно, с какими аргументами был вызов.

Пример:
```csharp
await repository.Received(1).SaveAsync(Arg.Is<Order>(x => x.UserId == userId));
```

---

## Что добавить про асинхронные тесты
Для `async`-тестов:
- всегда используй `async Task`, а не `async void`;
- `await` должен быть у целевого действия;
- исключения проверяй через асинхронные ассерты;
- не используй `.Result` и `.Wait()`.

Пример:
```csharp
Func<Task> act = () => handler.Handle(command, CancellationToken.None);

await act.Should().ThrowAsync<ValidationException>();
```

---

## Что добавить про стабильность и скорость
Тесты должны быть быстрыми. Если тест медленный, это должно быть оправдано его уровнем.

Правила:
- unit-тесты не должны ходить во внешнюю инфраструктуру;
- избегай лишнего IO;
- не используй реальные задержки;
- не создавай избыточно тяжелые графы объектов;
- общий тестовый набор должен быть пригоден для частого локального запуска.

---

## Что добавить про организацию файлов
Рекомендуемая структура:
- располагай тесты рядом по доменному смыслу;
- имя файла должно совпадать с тестируемым классом или сценарием;
- один крупный класс тестов лучше разбить, если он становится трудно читаемым;
- используйте единый стиль папок во всем тестовом проекте.

Пример:
- `Application/Users/Commands/CreateUser/CreateUserCommandHandlerTests.cs`
- `Application/Users/Validators/CreateUserCommandValidatorTests.cs`

---

## Что добавить про anti-patterns
Явно запрети следующие практики:

- Один тест проверяет сразу несколько независимых сценариев.
- Случайные данные без контроля и воспроизводимости.
- Скрытая логика в helper-методах.
- Избыточное использование моков.
- Проверка внутренних деталей вместо контракта.
- Большие нечитаемые `Arrange`-блоки без выноса повторяющегося кода.
- Ассерты без достаточного контекста.
- Отсутствие `summary`.
- Имена тестов вида `Test1`, `ShouldWork`, `ValidateTest`.
- Использование `DateTime.UtcNow` прямо в тесте.
- Хрупкие проверки строк целиком там, где достаточно проверить код ошибки или ключевую часть сообщения.

## Практический шаблон теста
```csharp
/// <summary>
/// Проверяет, что обработчик создает заказ, когда команда содержит валидные данные.
/// </summary>
[Fact]
public async Task Handle_ShouldCreateOrder_WhenCommandIsValid()
{
    // Arrange
    var userId = Guid.Parse("11111111-1111-1111-1111-111111111111");
    var command = new CreateOrderCommandBuilder()
        .WithUserId(userId)
        .Build();

    var repository = Substitute.For<IOrderRepository>();
    var handler = new CreateOrderCommandHandler(repository);

    // Act
    var result = await handler.Handle(command, CancellationToken.None);

    // Assert
    result.Should().NotBeNull();
    await repository.Received(1).SaveAsync(Arg.Is<Order>(x => x.UserId == userId));
}
```

## Чек-лист для каждого нового теста
Перед завершением работы проверь:

- Есть XML `summary` на русском.
- Имя теста соответствует шаблону.
- Тест покрывает один логический сценарий.
- Структура `Arrange / Act / Assert` сохранена.
- Нет лишних моков.
- Нет зависимости от времени, сети, файловой системы и случайности.
- Проверяется поведение, а не реализация.
- Нет дублирования, которое уже пора вынести в общий код.
- Assertions читаемые и диагностичные.
- Для сущности покрыты не только happy path, но и негативные и граничные кейсы.

# Test Writing Rules

## Goal
Писать тесты, которые легко читать, поддерживать, расширять и безопасно рефакторить без дублирования.

## Обязательное правило: summary на русском для каждого теста
Для каждого нового или измененного тестового метода добавляй XML `summary` на русском прямо над методом.

Требования:
- Описывает проверяемое поведение, а не внутреннюю реализацию.
- Кратко и конкретно: 1–2 предложения.
- Без общих формулировок вроде «Проверяет метод».
- Для `[Theory]` summary описывает общее поведение всех входных кейсов.

## Общие правила
- Используй структуру Arrange / Act / Assert.
- Используй понятные имена тестов в формате `Method_ShouldExpectedResult_WhenCondition`.
- Один тест должен проверять один логический сценарий.
- Тесты должны быть детерминированными.
- Любые внешние зависимости изолируй через моки, стабы или fake-реализации.
- Проверяй наблюдаемое поведение, а не детали реализации.
- Не дублируй `Arrange`-блоки более чем в 2 тестах.
- Не скрывай смысл теста в helper-методах.
- Используй выразительные и диагностичные assertions.

## Рекомендуемые библиотеки
- `xUnit`
- `FluentAssertions`
- `NSubstitute` или `Moq`
- `AutoFixture`
- `[Theory]` + `MemberData` / `ClassData`

## Переиспользуемый код
Выноси общий код в:
- `Application/Common/Fixtures`
- `Application/Common/Builders`
- `Application/Common/Fakes`
- `Application/Common/TestData`
- `Application/Common/Extensions`

## Минимальное покрытие сценариев
Для каждого юзкейса, валидатора или обработчика покрывай:
- позитивный сценарий;
- негативные сценарии;
- граничные значения;
- ошибки зависимостей;
- наборы входных данных через `[Theory]`, где это уместно.

## Дополнительно для validator tests
Проверяй:
- успешную валидацию валидного объекта;
- ошибки для каждого нарушенного правила;
- корректное поле, код и сообщение ошибки, если это часть контракта;
- поведение на `null`, пустых и граничных значениях.

## Дополнительно для async-тестов
- Используй `async Task`, а не `async void`.
- Не используй `.Result` и `.Wait()`.
- Проверяй исключения через асинхронные assertions.

## Anti-patterns
- Несколько независимых сценариев в одном тесте.
- Случайные нерепродуцируемые данные.
- Проверка внутренних деталей реализации.
- Избыточное количество моков.
- Хрупкие assertions.
- Отсутствие `summary`.
