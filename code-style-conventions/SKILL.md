# Code Style Conventions

## Цель
Зафиксировать единые соглашения по стилю кода, чтобы:
- ускорять ревью;
- снижать количество дефектов;
- улучшать читаемость и сопровождаемость;
- обеспечивать предсказуемое поведение кода в production.

## Когда использовать
Используй этот skill, если нужно:
- написать новый код в любом слое;
- отрефакторить существующий код;
- провести PR-ревью на соответствие стандартам;
- принять решение по спорному стилистическому вопросу.

## Обязательные правила
1. Использовать primary constructor для внедрения зависимостей там, где это возможно и уместно.
2. Ограничивать длину строки 120 символами.
3. Для каждого кастомного бизнес-сценария ошибки использовать отдельный exception-класс.
4. Писать тексты логов только на английском языке.
5. Писать тексты ошибок только на английском языке.
6. Не использовать общий `Exception` для ожидаемых бизнес-сценариев.
7. Всегда пробрасывать `CancellationToken` в async-цепочке, где он доступен.

## Рекомендуемые практики
1. Соблюдать SRP: один класс и один метод должны иметь одну ответственность.
2. Использовать имена, отражающие намерение, без неочевидных сокращений.
3. Предпочитать guard clauses вместо глубокой вложенности условий.
4. Держать методы короткими и фокусными по смыслу.
5. Избегать неявных побочных эффектов и скрытых зависимостей.
6. Предпочитать простые решения по KISS и не добавлять лишнюю абстракцию по YAGNI.
7. Убирать дублирование, если оно стабильное и повторяется более одного раза.

## Анти-паттерны
1. God-class с множеством несвязанных обязанностей.
Почему плохо: растет сложность, падает тестируемость.
Как лучше: разделять по сценариям и ответственности.

2. Длинные, трудно читаемые строки и выражения.
Почему плохо: повышается риск ошибок и сложность ревью.
Как лучше: переносить выражения и декомпозировать логику.

3. Бизнес-ошибки через общий `Exception`.
Почему плохо: теряется семантика и предсказуемая обработка.
Как лучше: создавать отдельные типизированные exceptions.

4. Логи и ошибки на смешанном языке.
Почему плохо: сложнее анализировать инциденты и поддерживать единый стиль.
Как лучше: использовать только английский язык для log/error messages.

5. Неиспользование `CancellationToken` в цепочке вызовов.
Почему плохо: ухудшается управляемость операций под нагрузкой.
Как лучше: передавать токен во все поддерживаемые async-вызовы.

## Примеры

### Do
```csharp
public class UpdateUserFullNameCommandHandler(
	IUsersRepository usersRepository) : ICommandHandler<UpdateUserFullNameCommand, Unit>
{
	public async Task<Unit> Handle(UpdateUserFullNameCommand request, CancellationToken ct)
	{
		await usersRepository.UpdateFullNameAsync(request.UserId, request.FullName, ct);
		return Unit.Value;
	}
}
```

### Don't
```csharp
public class Handler : ICommandHandler<UpdateUserFullNameCommand, Unit>
{
	private readonly IUsersRepository _repo;

	public Handler(IUsersRepository repo)
	{
		_repo = repo;
	}

	public async Task<Unit> Handle(UpdateUserFullNameCommand r, CancellationToken ct)
	{
		await _repo.UpdateFullNameAsync(r.UserId, r.FullName, CancellationToken.None);
		throw new Exception("Ошибка обновления пользователя");
	}
}
```

## Алгоритм принятия решений
1. Определи, это правило обязательное или рекомендованное.
2. Проверь, поддерживает ли тип конструкции primary constructor.
3. Оцени длину строк и читаемость метода.
4. Для ошибок выбери конкретный exception-класс, а не общий `Exception`.
5. Для логов и сообщений ошибок проверь язык: только английский.
6. Для async-цепочки проверь, что `CancellationToken` проброшен до нижнего слоя.
7. Если правило нарушено, исправь код до завершения задачи.

## Проверка перед завершением
1. Зависимости внедрены через primary constructor, где это возможно.
2. Нет строк длиннее 120 символов.
3. Нет общего `Exception` для бизнес-сценариев.
4. Все log/error messages на английском.
5. `CancellationToken` передается по async-цепочке.
6. Код читается без дополнительных пояснений и лишних комментариев.
7. Изменения не ухудшают существующую тестируемость.
