# Summary

## Цель
Задать единый стандарт XML-документации, чтобы:
- код был самодокументируемым для команды;
- IntelliSense и Quick Documentation в Rider отображались корректно и полно;
- публичный и внутренний API оставался понятным без чтения реализации;
- ревью и сопровождение проходили быстрее.

## Когда использовать
Используй этот skill, если нужно:
- создать новый DTO, класс, метод, интерфейс или enum;
- обновить сигнатуру метода, контракт DTO или поведение класса;
- провести PR-ревью качества XML-комментариев;
- подготовить код к стабильной поддержке и передаче в команду.

## Обязательные правила
1. Все DTO должны иметь XML `summary`.
2. Все классы должны иметь XML `summary`.
3. Все методы должны иметь XML `summary`.
4. Все тексты `summary` должны быть на русском языке.
5. `summary` должен быть исчерпывающим: описывать назначение, ожидаемое поведение и результат.
6. `summary` должен быть оформлен в стандартном XML-формате `/// <summary>...</summary>`, чтобы Rider корректно отображал и сворачивал комментарий.
7. Для методов и свойств-реализаций интерфейсов, а также для `override`-членов по умолчанию использовать `/// <inheritdoc/>` вместо дублирования текста.
8. Для методов обязательно добавлять `param` для каждого параметра.
9. Для методов с возвращаемым значением обязательно добавлять `returns`.
10. Для методов, которые могут бросать ожидаемые исключения, добавлять `exception`.
11. Описание должно соответствовать фактическому поведению кода и обновляться при изменении логики.

## Рекомендуемые практики
1. Начинать `summary` с одного короткого предложения о назначении, затем добавлять детали.
2. Для сложных сценариев использовать 2-4 предложения вместо одного перегруженного.
3. Использовать единый стиль формулировок: без разговорной речи и двусмысленности.
4. Для DTO описывать бизнес-смысл объекта и ключевых полей.
5. Для классов описывать ответственность и границы применения.
6. Для методов описывать вход, действие и результат в терминах контракта.
7. Для асинхронных методов явно указывать, что операция выполняется асинхронно.
8. Для nullable-параметров и nullable-результатов явно пояснять допустимость `null`.
9. Не дублировать имя типа или метода в `summary` без добавления смысла.
10. Сохранять длину строки до 120 символов и избегать избыточной разметки внутри XML.
11. Если реализация полностью следует контракту интерфейса/базового типа, предпочитать `inheritdoc`; отдельный `summary` писать только при дополнительном поведении.

## Анти-паттерны
1. Пустой или шаблонный `summary`.
Почему плохо: не дает контекста и не помогает в IDE.
Как лучше: описывать назначение и наблюдаемое поведение.

2. `summary`, который противоречит реализации.
Почему плохо: вводит в заблуждение и создает дефекты на интеграции.
Как лучше: обновлять комментарии вместе с кодом в том же change set.

3. Описание на английском или смешанном языке.
Почему плохо: нарушает стандарт команды.
Как лучше: писать `summary` только на русском языке.

4. Комментарий в виде обычного `//` или `/* */` вместо XML.
Почему плохо: Rider не показывает это как API-документацию и не сворачивает как XML docs.
Как лучше: использовать `/// <summary>...</summary>`.

5. Отсутствие `param`, `returns`, `exception` у методов, где они нужны.
Почему плохо: контракт метода неполный.
Как лучше: документировать параметры, результат и ожидаемые исключения явно.

6. Дублирование текста контракта в каждой реализации вместо наследования.
Почему плохо: появляются расхождения между интерфейсом и реализацией при изменениях.
Как лучше: использовать `/// <inheritdoc/>` в реализации и держать полный контракт в интерфейсе/базовом типе.

## Примеры

### Do
```csharp
/// <summary>
/// Обновляет полное имя существующего пользователя.
/// Выполняет операцию асинхронно и сохраняет изменения в хранилище.
/// </summary>
/// <param name="userId">Идентификатор пользователя, для которого выполняется обновление.</param>
/// <param name="fullName">Новое полное имя пользователя в нормализованном формате.</param>
/// <param name="ct">Токен отмены асинхронной операции.</param>
/// <returns>Задача, завершающаяся после успешного сохранения изменений.</returns>
/// <exception cref="UserNotFoundException">
/// Выбрасывается, если пользователь с указанным идентификатором не найден.
/// </exception>
public async Task UpdateFullNameAsync(Guid userId, string fullName, CancellationToken ct = default)
{
    // ...
}
```

```csharp
/// <inheritdoc/>
public async Task UpdateFullNameAsync(Guid userId, string fullName, CancellationToken ct = default)
{
    // ...
}
```

### Don't
```csharp
// Update user name
public async Task UpdateFullNameAsync(Guid userId, string fullName, CancellationToken ct = default)
{
    // ...
}
```

## Алгоритм принятия решений
1. Определи артефакт: DTO, класс, метод, интерфейс или enum.
2. Убедись, что у артефакта есть `summary` на русском языке.
3. Для метода опиши: что принимает, что делает, что возвращает.
4. Добавь `param` для всех параметров и `returns` для результата.
5. Если есть ожидаемое исключение, добавь `exception` с условием выброса.
6. Для реализаций интерфейса/override проверь, что используется `inheritdoc`, если поведение не расширено.
7. Проверь, что `summary` не противоречит текущей реализации.
8. Проверь отображение в Rider Quick Documentation и корректное сворачивание XML-комментария.

## Проверка перед завершением
1. Все DTO имеют `summary`.
2. Все классы имеют `summary`.
3. Все методы имеют `summary`.
4. Все `summary` написаны на русском языке.
5. У всех методов заполнены `param`, `returns` и `exception` там, где это применимо.
6. Описания полные, точные и соответствуют фактическому поведению.
7. XML-комментарии оформлены через `///` и корректно отображаются в Rider.
8. Нет пустых, формальных или устаревших комментариев.
9. Для реализаций интерфейсов и override-членов используется `inheritdoc`, где это применимо.
