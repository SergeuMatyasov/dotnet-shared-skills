# Controllers Testing

## Связанные skills
- `.github/skills/validators/SKILL.md`

## Цель
Гарантировать, что изменения в контроллерах не ломают контракт и поведение API.

## Когда использовать
Используй этот skill, если нужно:
- добавить/обновить тесты для endpoint;
- проверить совместимость между версиями API;
- закрыть регрессионный риск после рефакторинга контроллера.

## Обязательные правила
1. Для каждого измененного endpoint должен быть тест успешного сценария.
2. Для каждого измененного endpoint должен быть тест негативного сценария.
3. Проверять корректный статус-код и тип результата.
4. Проверять проброс `CancellationToken` в зависимости.
5. При versioning-изменениях проверять, что старая версия не сломана.

## Рекомендуемые практики
1. Тестировать согласованность `route id` и `body.UserId`, если это часть контракта.
2. Проверять тело ответа и headers/cookies, если они формируют контракт.
3. Добавлять regression-тесты для ранее обнаруженных дефектов.
4. Группировать тесты по endpoint и версии API.

## Анти-паттерны
1. Проверка только happy path без негативных веток.
2. Проверка внутренних деталей реализации вместо внешнего контракта.
3. Отсутствие тестов совместимости при добавлении новой версии.

## Примеры

### Минимальный набор на endpoint
1. Success test.
2. Validation/BadRequest test.
3. Dependency invocation test.
4. CancellationToken forwarding test.

## Алгоритм принятия решений
1. Определи, какие endpoint и версии затронуты.
2. Составь матрицу сценариев: success, validation, not found, auth, conflict.
3. Добавь/обнови тесты на контракт и коды ответа.
4. Добавь compatibility/regression тесты при необходимости.
5. Запусти тесты и проверь стабильность старой версии.

## Проверка перед завершением
1. Все измененные endpoint покрыты ключевыми сценариями.
2. Контрактные коды ответа проверены тестами.
3. Старая версия API не регрессировала.
4. Тесты проходят стабильно.
