# Segmented Change Review

## Цель
Сделать ревью больших изменений управляемым: показывать изменения блоками и получать подтверждение пользователя по каждому блоку.

## Когда использовать
Используй этот skill, если:
- в PR много файлов и нужен пошаговый разбор;
- пользователь хочет подтверждать изменения частями;
- нужно снизить риск пропустить регрессии.

## Принцип
1. Разбор выполняется только по логическим сегментам diff.
2. Не предлагать и не обсуждать версионирование API.
3. Не оценивать изменения через призму v1/v2/v3.

## Формат работы
1. Собери полный diff и сгруппируй изменения в логические сегменты.
2. Покажи сегменты списком с коротким названием и смыслом.
3. Разбирай сегменты по одному, в порядке риска: сначала критичные.
4. После каждого сегмента запрашивай решение пользователя: approve, revise, skip.
5. Переходи к следующему сегменту только после явного решения.

## Правила сегментации
1. Один сегмент = одна бизнес-идея или одна техническая цель.
2. Не смешивай в одном сегменте: API-контракт, доменную логику и инфраструктурные правки.
3. Переименования и форматирование выноси в отдельный сегмент.
4. Если сегмент слишком большой, дели по границам слоя: WebApi, Application, Infrastructure, Tests.

## Карточка сегмента
Для каждого сегмента показывай:
1. Что меняется.
2. Зачем это нужно.
3. Какие файлы затронуты.
4. Риски и возможные регрессии.
5. Что проверить тестами.
6. Рекомендация: approve или revise.

## Шаблон ответа
- Segment: <номер> / <название>
- Scope: <кратко>
- Files: <список>
- Behavior change: <есть/нет + кратко>
- Risks: <список>
- Validation: <какие тесты/проверки>
- Recommendation: <approve/revise>
- Decision required: approve <номер> | revise <номер> | skip <номер>

## Критерии качества
1. Все изменения отнесены к сегментам без пропусков.
2. Для каждого сегмента описаны риски и проверки.
3. Нет продвижения дальше без решения пользователя.
4. Итоговый сводный список принятых и отклоненных сегментов сформирован.

## Ограничения
1. Skill задает процесс и стиль ответа, но не заменяет инженерную валидацию.
2. Для критичных изменений обязательно подтверждай запуск тестов/проверок.
3. Не вводи рекомендации про API versioning.
