# Logs

## Цель
Задать единый стандарт логирования, который:
- помогает быстро диагностировать инциденты;
- не создает шум в production;
- безопасен по данным;
- пригоден для поиска, агрегации и алертинга.

## Когда использовать
Используй этот skill, если нужно:
- добавить новые логи в коде;
- изменить существующие сообщения или уровни логирования;
- провести PR-ревью логирования;
- стандартизировать логирование в новом модуле.

## Обязательные правила
1. Все тексты логов должны быть на английском языке.
2. Использовать structured logging: message template + параметры.
3. Не использовать string interpolation и конкатенацию в log message.
4. Для ошибок обязательно передавать exception object, если он есть.
5. Не логировать секреты, токены, пароли, OTP-коды, персональные данные в открытом виде.
6. Уровень логирования должен соответствовать серьезности события.
7. Логи должны быть полезными для расследования инцидента, а не декоративными.
8. Длина строк в коде не должна превышать 120 символов.

## Рекомендуемые практики
1. Добавлять стабильный контекст: `UserId`, `OrderId`, `SessionId`, `RequestId`.
2. Использовать унифицированные глаголы действий: `Created`, `Updated`, `Deleted`, `Failed`.
3. Писать сообщения в present/past tense без разговорных формулировок.
4. Логировать ключевые бизнес-шаги, а не каждую внутреннюю операцию.
5. Использовать события начала/завершения только для реально долгих или критичных операций.
6. Добавлять correlation id в каждый лог запроса, где это поддерживается.
7. Для retry-сценариев логировать номер попытки и итоговый исход.
8. Для внешних интеграций логировать провайдера, операцию, latency и результат.

## Уровни логирования
1. `Trace`: детальная диагностическая информация для локальной отладки.
2. `Debug`: технические детали выполнения, полезные разработчику.
3. `Information`: важные штатные события бизнес-потока.
4. `Warning`: отклонение от нормы, но сценарий продолжает работать.
5. `Error`: операция завершилась ошибкой, требуется внимание.
6. `Critical`: системная деградация или потеря ключевой функциональности.

## Анти-паттерны
1. Слишком общие сообщения.
Плохо: `"Something went wrong"`.
Лучше: `"Failed to update user email for {UserId}"`.

2. Логирование без контекста.
Плохо: `"Validation failed"`.
Лучше: `"Validation failed for {CommandName} with {ErrorCount} errors"`.

3. Неверный уровень логирования.
Плохо: штатный бизнес-отказ в `Error`.
Лучше: `Warning` для ожидаемого отказа и `Error` для реальной ошибки.

4. Дублирование одинаковых логов в одной цепочке.
Плохо: логирование одной и той же ошибки на каждом уровне стека.
Лучше: один подробный лог в месте принятия решения об обработке.

5. Утечка чувствительных данных.
Плохо: `"Token: {Token}"`.
Лучше: `"Token was provided"`.

## Примеры

### Do
```csharp
logger.LogInformation(
	"Updated full name for user {UserId}",
	userId);

logger.LogError(
	exception,
	"Failed to refresh token for session {SessionId}",
	sessionId);
```

### Don't
```csharp
logger.LogInformation($"User {userId} full name updated");
logger.LogError("Something went wrong");
logger.LogDebug("JWT token: {Token}", token);
```

## Алгоритм принятия решений
1. Определи цель лога: диагностика, аудит, бизнес-событие, ошибка.
2. Выбери уровень по влиянию на систему и клиента.
3. Сформируй короткий message template с глаголом действия.
4. Добавь минимально достаточный контекст для расследования.
5. Проверь, что в параметрах нет чувствительных данных.
6. Убедись, что лог не дублирует соседние логи в этом же потоке.
7. Проверь, что сообщение на английском и строка кода <= 120 символов.

## Проверка перед завершением
1. Все новые и измененные сообщения логов на английском языке.
2. Нет interpolation/конкатенации в log message.
3. Для ошибок передается `exception` и достаточный контекст.
4. Нет утечки секретов и персональных данных.
5. Выбранный уровень логирования соответствует серьезности события.
6. Логи позволяют восстановить цепочку событий без чтения кода.
7. Нет дублирующих шумных логов в горячих участках.
