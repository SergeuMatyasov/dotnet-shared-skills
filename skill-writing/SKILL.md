# Skill Writing

## Цель

Этот skill используется при создании, редактировании и ревью любых внутренних SKILL-файлов.

Его задача:
- делать skills понятными для людей и LLM-инструментов,
- обеспечивать единый формат,
- снижать двусмысленность,
- повышать качество инструкций для Codex/Copilot,
- предотвращать расплывчатые, противоречивые и бесполезные правила.

Skill должен быть написан так, чтобы:
1. человек быстро понял, когда его применять;
2. модель поняла, что делать;
3. правила можно было проверить на ревью;
4. примеры помогали, а не дублировали абстракции.

---

## Когда использовать

Используй этот skill, если нужно:
- создать новый skill;
- дополнить существующий skill;
- переписать неструктурированный skill;
- сделать skill более строгим и прикладным;
- провести ревью качества skill-файла;
- унифицировать несколько skills в одном стиле.

---

## Основные принципы

### 1. Skill должен быть практическим
Skill пишется не ради описания темы, а ради применения в работе.

Хорошо:
- описывает, когда skill использовать;
- задает конкретные правила;
- содержит критерии качества;
- показывает хорошие и плохие примеры;
- дает проверяемый checklist.

Плохо:
- состоит из общих рассуждений;
- повторяет очевидности;
- не говорит, как принимать решение;
- не содержит проверяемых требований.

---

### 2. Skill должен быть однозначным
Формулировки должны быть конкретными и проверяемыми.

Хорошо:
- "Все тексты логов должны быть на английском языке."
- "Не использовать string interpolation в log message, если доступен structured logging."
- "Для ошибок логировать исключение и контекст операции."

Плохо:
- "Писать хорошие логи."
- "Стараться делать сообщения понятными."
- "По возможности использовать правильный уровень."

Если правило нельзя проверить на PR review, оно слишком расплывчатое.

---

### 3. Skill должен быть ориентирован на решение
Каждый раздел должен отвечать хотя бы на один из вопросов:
- Когда это применять?
- Что делать?
- Чего не делать?
- Как выбрать между вариантами?
- Как проверить результат?

---

### 4. Skill должен разделять обязательные правила и рекомендации
Не смешивай жесткие требования и полезные советы в одном списке.

Используй явное разделение:
- обязательные правила;
- рекомендуемые практики;
- анти-паттерны;
- checklist перед завершением.

Это снижает путаницу.

---

### 5. Skill должен учитывать контекст команды
Язык, терминология и примеры должны соответствовать среде, где skill реально используется.

Для внутренней русскоязычной команды допустимо:
- пояснения писать на русском;
- доменные примеры брать из реального стека команды;
- оставлять технические термины в оригинале, если так принято в кодовой базе.

Если skill задает правила для артефактов, которые должны быть на английском, это нужно явно фиксировать как требование.

---

## Рекомендуемая структура SKILL-файла

Базовая структура:

```md
---
name: skill-name
description: "Краткое описание, когда использовать этот skill."
---

# Skill Name

## Цель

## Когда использовать

## Обязательные правила

## Рекомендуемые практики

## Анти-паттерны

## Примеры

## Алгоритм принятия решений

## Проверка перед завершением
```

Не все разделы обязательны во всех skills, но структура должна оставаться логичной и предсказуемой.

---

## Требования к header

### Поле `name`
Должно быть:
- коротким;
- стабильным;
- понятным;
- в kebab-case или другом одном принятом формате.

Хорошо:
- `logs`
- `skill-writing`
- `api-design`
- `pull-request-review`

Плохо:
- `My Skill`
- `SomeRulesForLogging`
- `skill for logs and tracing and metrics`

---

### Поле `description`
Должно кратко объяснять:
- когда использовать skill;
- для какой задачи он нужен.

Хорошо:
- `"Используй при добавлении и ревью логирования: язык логов, структура сообщений и консистентность контекста."`
- `"Используй при создании и ревью DTO: контракты, nullable, naming и backward compatibility."`

Плохо:
- `"Полезный skill."`
- `"Разные правила и советы."`

---

## Требования к стилю написания

### 1. Писать коротко и плотно
Каждый пункт должен нести смысловую нагрузку.

Хорошо:
- "Не логировать токены, пароли и секреты в открытом виде."

Плохо:
- "Важно помнить, что в современном мире разработки необходимо всегда внимательно относиться к чувствительным данным."

---

### 2. Избегать воды
Не писать длинные вступления, если они не добавляют критериев принятия решений.

---

### 3. Использовать повелительные и нормативные формулировки
Для обязательных правил использовать ясные конструкции:
- "Должно"
- "Не должно"
- "Использовать"
- "Не использовать"
- "Обязательно"
- "Запрещено"

Для рекомендаций:
- "Предпочтительно"
- "Рекомендуется"
- "Обычно"
- "Допустимо"

---

### 4. Не смешивать норму и объяснение
Сначала правило, потом при необходимости короткое объяснение.

Хорошо:
- "Не использовать string interpolation в логах. Это ухудшает structured search и консистентность полей."

Плохо:
- длинный абзац, внутри которого неясно, что именно запрещено.

---

## Как формулировать правила

Хорошее правило обладает четырьмя свойствами:
1. конкретно;
2. проверяемо;
3. применимо;
4. полезно в типовом сценарии.

Примеры хороших правил:
- "Все публичные API DTO должны иметь явные nullable-контракты."
- "Имена feature flag должны отражать бизнес-смысл, а не временную реализацию."
- "В логах ошибок обязательно передавать exception object, если он есть."

Примеры плохих правил:
- "Код должен быть красивым."
- "Нужно стараться писать качественно."
- "Лучше делать удобно."

---

## Как писать примеры

Примеры обязательны, если без них правило можно понять двояко.

Каждый пример должен быть одного из типов:
- хорошо / плохо;
- do / don't;
- before / after;
- допустимо / недопустимо.

Примеры должны:
- быть короткими;
- показывать суть правила;
- использовать знакомый стек команды;
- не быть игрушечными без необходимости.

---

## Как писать анти-паттерны

Анти-паттерны нужны, если команда регулярно допускает повторяющиеся ошибки.

Формат:
- назвать проблему;
- кратко объяснить, почему это плохо;
- показать корректную альтернативу.

Пример:
- "Слишком общие описания. Skill с формулировками вроде 'пиши лучше код' не помогает модели принять решение."
- "Смешение обязательных правил и опциональных советов. Это делает skill непредсказуемым при применении."
- "Отсутствие примеров. Из-за этого инструкция становится декларативной, но неоперационной."

---

## Как писать checklist

Checklist должен быть коротким и проверяемым.

Он нужен для финальной самопроверки.

Хороший checklist:
1. Название skill короткое и понятное.
2. Description объясняет, когда применять skill.
3. Обязательные правила конкретны и проверяемы.
4. Есть примеры для неоднозначных мест.
5. Нет противоречий между разделами.
6. Нет дублирования одинаковых правил.
7. Skill ориентирован на реальные решения, а не на общую теорию.

Плохой checklist:
- "Все хорошо."
- "Текст нормальный."
- "Skill полезный."

---

## Алгоритм написания нового skill

### Шаг 1. Определи границы
Сформулируй:
- для какой задачи skill;
- где он применяется;
- что он должен улучшать;
- что не входит в его scope.

Пример:
- skill про логи не должен превращаться в полный observability handbook;
- skill про DTO не должен описывать весь API design.

---

### Шаг 2. Выдели типовые решения
Определи, какие решения человек или модель должны принимать с помощью этого skill.

Например:
- какой уровень логирования выбрать;
- как назвать DTO;
- как оформить retry;
- когда использовать async.

---

### Шаг 3. Сформулируй обязательные правила
Оставь только те правила, которые:
- реально важны;
- часто нарушаются;
- дают единообразие;
- могут быть проверены.

---

### Шаг 4. Добавь рекомендации
После обязательных правил добавь полезные практики, которые улучшают результат, но не всегда обязательны.

---

### Шаг 5. Добавь анти-паттерны
Опиши самые частые ошибки, чтобы skill не только направлял, но и предотвращал деградацию.

---

### Шаг 6. Добавь примеры
Особенно там, где:
- возможны разные трактовки;
- важно соблюдение формата;
- требуется стиль или шаблон.

---

### Шаг 7. Добавь checklist
Финальная секция должна позволять быстро проверить качество skill.

---

## Критерии хорошего skill

Хороший skill:
- узкий по scope;
- конкретный;
- полезный в PR и генерации кода;
- не противоречивый;
- не раздутый;
- содержит применимые правила;
- помогает принимать решения;
- одинаково полезен человеку и модели.

---

## Критерии плохого skill

Плохой skill:
- слишком общий;
- состоит из банальностей;
- не содержит обязательных норм;
- не имеет примеров;
- противоречит сам себе;
- смешивает правила, советы и философию;
- не дает критериев проверки;
- описывает тему, а не рабочее поведение.

---

## Anti-patterns

### 1. Слишком широкий scope
Плохо:
- один skill одновременно про coding style, architecture, tests, logs и API contracts.

Почему плохо:
- модель не понимает приоритеты;
- правила начинают конфликтовать;
- skill трудно применять.

Как лучше:
- делать отдельные skills по темам.

---

### 2. Повторение общих инженерных истин
Плохо:
- "Пиши качественный код."
- "Думай о поддерживаемости."
- "Старайся делать лучше."

Почему плохо:
- не помогает выбрать конкретное действие.

Как лучше:
- писать конкретные критерии и шаблоны решений.

---

### 3. Отсутствие границ применения
Плохо:
- неясно, когда skill нужно использовать, а когда нет.

Как лучше:
- явно добавлять раздел "Когда использовать".

---

### 4. Примеры без правил
Плохо:
- много примеров, но нет явных норм.

Как лучше:
- сначала правило, потом пример.

---

### 5. Правила без примеров там, где есть неоднозначность
Плохо:
- правило можно понять по-разному.

Как лучше:
- добавить короткие do/don't examples.

---

### 6. Конфликт между разделами
Плохо:
- в одном месте сказано "всегда", в другом "по возможности".

Как лучше:
- развести обязательное и рекомендованное.

---

```md

description: "Используй при <сценарий применения>."
---

## Цель
Кратко опиши, какую проблему решает этот skill и какой результат должен обеспечивать.

## Когда использовать
Перечисли сценарии, в которых этот skill должен применяться.

## Обязательные правила
1. ...
2. ...
3. ...

## Рекомендуемые практики
1. ...
2. ...
3. ...

## Анти-паттерны
1. ...
2. ...
3. ...

## Примеры
### Хорошо
...
### Плохо
...

1. ...

1. ...
```

## Правила языка

Для внутренней русскоязычной команды:
- текст skill допустимо писать на русском;
- код, идентификаторы, API names и технические термины оставлять как принято в кодовой базе;
- если skill регулирует артефакт с обязательным языком, это нужно фиксировать явно.

Пример:
- skill можно писать на русском;
- но если это skill про логирование, нужно явно указать, что сами log messages должны быть только на английском.

---

## Правила для skills, которые управляют форматом артефактов

Если skill задает требования к конкретному типу артефактов, он должен явно описывать:
1. формат;
2. язык;
3. обязательные поля;
4. запрещенные практики;
5. критерии качества;
6. примеры.

Например:
- для логов: язык, структура, уровень, контекст, безопасность;
- для DTO: naming, nullability, backward compatibility;
- для тестов: deterministic behavior, naming, assertions, isolation;
- для PR review: severity, evidence, actionability, tone.

---

## Проверка перед завершением

1. У skill есть понятный `name`.
2. `description` объясняет, когда skill применять.
3. Scope skill ограничен и не размыт.
4. Обязательные правила конкретны и проверяемы.
5. Рекомендации отделены от обязательных правил.
6. Есть анти-паттерны для частых ошибок.
7. Есть примеры для неоднозначных случаев.
8. Нет противоречий между разделами.
9. Нет пустых или банальных формулировок.
10. Skill помогает принять рабочее решение в IDE, а не просто описывает тему.

---

## Definition of done

Skill считается готовым, если:
- его можно отдать другому инженеру без устных пояснений;
- по нему можно делать PR review;
- по нему модель сможет генерировать более консистентный результат;
- спорные места сведены к минимуму;
- правила не требуют "догадки", что автор имел в виду.
