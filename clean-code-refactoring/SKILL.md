# Clean Code Refactoring

## Цель
Выполнять безопасный рефакторинг, улучшая структуру кода без изменения внешнего поведения.

## Когда использовать
Используй этот skill, если нужно:
- уменьшить сложность существующего кода;
- подготовить код к расширению;
- устранить дублирование и технический долг.

## Обязательные правила
1. Рефакторинг не должен менять внешний контракт без явной задачи.
2. Изменения делать небольшими проверяемыми шагами.
3. После каждого логического шага сохранять компилируемость и работоспособность.
4. Не смешивать крупный рефакторинг с несвязанными функциональными изменениями.

## Рекомендуемые практики
1. Сначала покрыть рискованные зоны тестами.
2. Использовать safe rename через инструменты IDE.
3. Убирать дублирование после стабилизации общего паттерна.
4. Выделять сложные блоки в отдельные функции/классы.
5. Фиксировать причину и цель рефакторинга в описании PR.

## Анти-паттерны
1. Big-bang рефакторинг без промежуточной проверки.
2. Переименование и функциональные изменения в одном неразделенном коммите.
3. Рефакторинг без проверки совместимости публичного контракта.

## Алгоритм принятия решений
1. Зафиксируй текущее поведение и риски.
2. Разбей рефакторинг на маленькие шаги.
3. Выполняй шаг, проверяй сборку и тесты.
4. Повторяй до достижения целевой структуры.
5. Проверь отсутствие регрессий.

## Проверка перед завершением
1. Внешнее поведение сохранено.
2. Код стал проще и читабельнее.
3. Нет новых архитектурных связностей.
4. Сборка и релевантные тесты проходят.
