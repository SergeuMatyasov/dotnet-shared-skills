# Errors

## Цель
Сформировать единый стандарт ошибок, который:
- предсказуем для клиента API;
- удобен для диагностики и поддержки;
- безопасен по данным;
- снижает вероятность регрессий при развитии кода.

## Когда использовать
Используй этот skill, если нужно:
- добавить новый exception-класс;
- изменить обработку ошибок в service/handler/controller;
- провести PR-ревью error handling;
- стандартизировать формат ошибок в API.

## Обязательные правила
1. Для каждого кастомного бизнес-сценария использовать отдельный exception-класс.
2. Не использовать общий `Exception` для expected бизнес-ошибок.
3. Все тексты ошибок должны быть на английском языке.
4. Публичные error messages должны быть понятными и без утечки внутренней реализации.
5. Expected ошибки маппить в предсказуемые HTTP-коды через единый error handling слой.
6. Для unexpected ошибок не раскрывать детали клиенту, но логировать их на сервере.
7. Сохранять причинно-следственную связь через `innerException`, если это важно.
8. Длина строк в коде не должна превышать 120 символов.

## Рекомендуемые практики
1. Называть исключения по шаблону: `<Condition>Exception` или `<Domain><Condition>Exception`.
2. Использовать fail-fast: валидировать вход до выполнения побочных эффектов.
3. Делать сообщение ошибки коротким, однозначным и в нейтральном тоне.
4. Разделять технические детали (для логов) и клиентское сообщение (для ответа API).
5. Для часто повторяемых сценариев создавать переиспользуемые базовые типы исключений.
6. Для валидационных ошибок возвращать единый формат с полем/причиной/кодом.
7. Документировать контракт ошибок в OpenAPI для внешних endpoint.

## Категории ошибок
1. `Validation errors`: некорректный входной запрос.
2. `Business rule violations`: нарушение доменных правил.
3. `Not found`: запрошенный ресурс отсутствует.
4. `Conflict`: конфликт состояния или версии.
5. `Unauthorized/Forbidden`: проблемы прав доступа.
6. `Infrastructure errors`: сеть, БД, внешние сервисы.
7. `Unexpected errors`: непредвиденные исключения.

## Анти-паттерны
1. Универсальный `throw new Exception("...")`.
Почему плохо: теряется семантика ошибки и корректный mapping в HTTP.
Как лучше: использовать специализированный exception-класс.

2. Сообщение ошибки на русском или смешанном языке.
Почему плохо: нарушается единый стандарт и анализ логов.
Как лучше: писать сообщения только на английском.

3. Утечка внутренних деталей в API response.
Почему плохо: риски безопасности и привязка клиента к внутренней реализации.
Как лучше: наружу отдавать безопасное сообщение, детали оставлять в логах.

4. Проглатывание исключений.
Почему плохо: появляются silent failures и сложная диагностика.
Как лучше: логировать и обрабатывать явно, либо rethrow.

5. Дублирование одинаковой обработки ошибок в каждом контроллере.
Почему плохо: несогласованность поведения и рост техдолга.
Как лучше: централизованный global exception handler.

## Примеры

### Do
```csharp
public sealed class UserNotFoundException(Guid userId)
	: Exception($"User with id '{userId}' was not found.")
{
}

if (!isValid)
{
	throw new InvalidAuthenticationRequestException("Authentication request is invalid.");
}
```

### Don't
```csharp
throw new Exception("Пользователь не найден");
throw new Exception($"DB connection failed: {connectionString}");
```

## Алгоритм принятия решений
1. Определи тип ошибки: validation, business, infrastructure или unexpected.
2. Реши, ожидаемая ли это ошибка в бизнес-потоке.
3. Для expected ошибки выбери специализированный exception-класс.
4. Сформулируй короткое английское сообщение без секретов и внутренних деталей.
5. Определи корректный HTTP mapping для API-слоя.
6. Убедись, что ошибка будет обработана централизованно.
7. Добавь или обнови тесты на критичный error-сценарий.

## Проверка перед завершением
1. Нет мест с общим `Exception` для бизнес-сценариев.
2. Все сообщения ошибок и валидации на английском.
3. Выбранный тип исключения соответствует одному сценарию.
4. Нет утечки секретов и внутренних деталей в публичных ответах.
5. Ошибки корректно обрабатываются global exception handler.
6. Для ключевых ошибок есть тесты или обновлена существующая проверка.
