# Clean Code SOLID

## Цель
Применять SOLID так, чтобы код был расширяемым, стабильным и безопасным к изменениям.

## Когда использовать
Используй этот skill, если нужно:
- проектировать новый компонент;
- рефакторить перегруженный класс;
- уменьшить связность между слоями.

## Обязательные правила
1. SRP: класс должен иметь одну причину для изменения.
2. OCP: расширять поведение без правок стабильного кода, где это уместно.
3. LSP: реализации не должны ломать контракт базового типа.
4. ISP: не заставлять клиентов зависеть от лишних методов интерфейса.
5. DIP: зависеть от абстракций, а не от конкретных реализаций.

## Рекомендуемые практики
1. Декомпозировать крупные сервисы на узкие интерфейсы.
2. Избегать циклических зависимостей между модулями.
3. Проверять жизненный цикл зависимостей в DI.
4. Выделять policy/strategy для вариативного поведения.

## Анти-паттерны
1. Интерфейс-комбайн с десятками несвязанных методов.
2. Нарушение LSP через `NotImplementedException` в рабочем контракте.
3. Жесткая зависимость application к инфраструктурной реализации.

## Алгоритм принятия решений
1. Определи ответственность компонента.
2. Проверь, не тянет ли он чужие обязанности.
3. Выдели абстракции на границах модулей.
4. Проверь заменяемость реализаций по контракту.

## Проверка перед завершением
1. Нет явного нарушения SRP/ISP.
2. Критические зависимости идут через абстракции.
3. Расширение сценариев не требует массового изменения стабильного кода.
