# Clean Code Naming

## Цель
Сделать имена в коде однозначными, выразительными и устойчивыми к рефакторингу.

## Когда использовать
Используй этот skill, если нужно:
- назвать новый класс, метод, поле, DTO или enum;
- переименовать существующие сущности;
- провести ревью качества naming.

## Обязательные правила
1. Имя должно отражать намерение и доменный смысл.
2. Запрещены неочевидные сокращения и технический шум в публичных именах.
3. Имена не должны содержать временный контекст реализации (`Temp`, `New`, `Data2`).
4. В одном контексте одно понятие должно иметь одно имя.
5. Публичные имена не должны быть двусмысленными (`Process`, `Handle`, `Do`).

## Рекомендуемые практики
1. Для методов использовать глагол действия (`Get`, `Update`, `Create`, `Delete`).
2. Для DTO использовать явные суффиксы по роли (`Request`, `Response`, `Dto`).
3. Для bool-значений использовать префиксы `Is`, `Has`, `Can`, `Should`.
4. Для коллекций использовать множественное число.
5. Для enum использовать значения, понятные без чтения комментариев.

## Анти-паттерны
1. `obj`, `data`, `item`, `model` в бизнес-логике.
2. Разные имена для одного и того же понятия в соседних слоях.
3. Имена с избыточной реализационной детализацией (`UserSqlRepositoryServiceImpl`).

## Примеры

### Do
- `UpdateUserFullNameCommand`
- `GetByIdAsync`
- `isUserActive`

### Don't
- `DoWork`
- `ProcessData`
- `tmpUser`

## Алгоритм принятия решений
1. Определи доменную роль сущности.
2. Сформируй имя по роли, а не по технической реализации.
3. Проверь, что имя однозначно читается в месте использования.
4. Проверь консистентность с соседними модулями.

## Проверка перед завершением
1. Имена отражают намерение.
2. Нет неочевидных сокращений.
3. Нет двусмысленных публичных имен.
4. Сохраняется единообразие терминов между слоями.
